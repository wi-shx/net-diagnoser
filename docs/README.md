## 🔥 5个常见网络问题场景

### 场景1: 连接超时

**背景**：
用户报告无法访问某个服务，浏览器一直转圈。

**症状**：
- 浏览器显示"连接超时"
- API调用失败，提示timeout
- 服务日志中出现大量超时错误

**日志示例**：
```
[15/Feb/2026:10:00:00 +0800] ERROR: Connection timeout to 192.168.1.100:8080
[15/Feb/2026:10:00:01 +0800] ERROR: Connection timeout to 192.168.1.100:8080
[15/Feb/2026:10:00:02 +0800] ERROR: Connection timeout to 192.168.1.100:8080
```

**常见原因**：
1. 服务器防火墙阻止连接
2. 目标服务未启动
3. 网络路由问题
4. 服务器负载过高

**排查思路**：
1. 检查防火墙规则（iptables -L -n）
2. 检查服务状态（systemctl status nginx）
3. 测试网络连通性（ping target-server.com）
4. 检查服务器负载（top, htop）

---

### 场景2: DNS解析失败

**背景**：
用户报告无法访问某个域名，但使用IP地址可以访问。

**症状**：
- 浏览器显示"DNS_PROBE_FINISHED_NXDOMAIN"
- 域名无法ping通，但IP可以
- 服务日志中出现DNS解析错误

**日志示例**：
```
[15/Feb/2026:10:00:00 +0800] ERROR: DNS resolution failed for api.example.com
[15/Feb/2026:10:00:01 +0800] ERROR: getaddrinfo failed for api.example.com
```

**常见原因**：
1. DNS服务器配置错误
2. DNS记录不存在或过期
3. 网络防火墙阻止DNS查询（UDP/53端口）
4. DNS服务器响应超时

**排查思路**：
1. 检查DNS配置（cat /etc/resolv.conf）
2. 测试DNS解析（dig api.example.com）
3. 检查DNS服务器连通性（ping 8.8.8.8）
4. 检查防火墙规则（iptables -L -n | grep 53）

---

### 场景3: 端口不可达

**背景**：
用户报告某个服务端口无法连接。

**症状**：
- telnet连接失败
- curl请求失败
- 服务日志中出现端口拒绝错误

**日志示例**：
```
[15/Feb/2026:10:00:00 +0800] ERROR: Connection refused to 192.168.1.100:8080
[15/Feb/2026:10:00:01 +0800] ERROR: Failed to connect to port 8080
```

**常见原因**：
1. 服务未监听该端口
2. 防火墙阻止该端口
3. 端口被其他进程占用
4. 服务配置错误

**排查思路**：
1. 检查端口监听状态（ss -tlnp | grep 8080）
2. 检查防火墙规则（iptables -L -n | grep 8080）
3. 检查服务配置（cat /etc/nginx/nginx.conf）
4. 测试端口连接（telnet 192.168.1.100 8080）

---

### 场景4: 高延迟

**背景**：
用户报告服务响应很慢，页面加载时间长。

**症状**：
- API调用延迟超过预期（>1秒）
- 页面加载时间过长
- 日志中出现响应时间过长记录

**日志示例**：
```
[15/Feb/2026:10:00:00 +0800] WARN: Response time exceeded threshold: 2.5s
[15/Feb/2026:10:00:01 +0800] WARN: Slow query detected: 3.1s
```

**常见原因**：
1. 网络延迟过高
2. 服务器负载过高
3. 数据库查询慢
4. 网络拥塞

**排查思路**：
1. 测试网络延迟（ping -c 10 target-server.com）
2. 检查服务器负载（top, htop）
3. 检查数据库查询（slow query log）
4. 追踪网络路径（traceroute target-server.com）

---

### 场景5: 服务异常

**背景**：
用户报告服务返回HTTP 5xx错误。

**症状**：
- API返回500/502/503错误
- 服务日志中出现异常堆栈
- 服务进程崩溃

**日志示例**：
```
[15/Feb/2026:10:00:00 +0800] ERROR: HTTP 500 Internal Server Error
[15/Feb/2026:10:00:01 +0800] ERROR: Database connection failed
[15/Feb/2026:10:00:02 +0800] FATAL: Out of memory
```

**常见原因**：
1. 代码bug
2. 数据库连接失败
3. 内存不足
4. 依赖服务故障

**排查思路**：
1. 检查服务状态（systemctl status nginx）
2. 查看服务日志（journalctl -u nginx -n 100）
3. 检查服务配置（cat /etc/nginx/nginx.conf）
4. 重启服务（systemctl restart nginx）

## 📊 场景优先级

| 场景 | 优先级 | 原因 |
|------|--------|------|
| 场景1: 连接超时 | P0 | 最常见的问题，影响范围广 |
| 场景2: DNS解析失败 | P0 | 影响所有域名访问 |
| 场景3: 端口不可达 | P0 | 服务不可用的高频问题 |
| 场景4: 高延迟 | P1 | 用户体验问题，但服务可用 |
| 场景5: 服务异常 | P1 | 通常不是网络问题，而是应用层问题 |
