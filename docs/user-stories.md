## 📋 用户故事详细说明

### US-001: 上传日志文件

**用户故事**：
> 作为运维工程师，我希望能上传日志文件，以便工具能自动分析

**验收标准**：
- [ ] 支持.log和.txt格式文件
- [ ] 单个文件最大支持100MB
- [ ] 文件不存在时给出友好提示
- [ ] 上传成功显示文件基本信息（大小、行数）

**测试用例**：
1. 上传有效的.log文件 → 成功
2. 上传有效的.txt文件 → 成功
3. 上传不存在的文件 → 错误提示
4. 上传超过100MB的文件 → 错误提示
5. 上传非文本文件（如图片） → 错误提示

---

### US-002: 查看错误统计

**用户故事**：
> 作为运维工程师，我希望能看到错误日志的统计信息，以便快速了解问题严重程度

**验收标准**：
- [ ] 显示总日志行数
- [ ] 显示错误行数
- [ ] 显示错误率（百分比）
- [ ] 按错误级别分组（如ERROR、WARN、FATAL）
- [ ] 按错误类型分组（如连接超时、DNS失败）

**测试用例**：
1. 无错误的日志文件 → 显示0错误率
2. 纯错误日志文件 → 显示100%错误率
3. 混合日志文件 → 显示正确统计
4. 空文件 → 显示0行日志

---

### US-003: 获取AI分析结果

**用户故事**：
> 作为运维工程师，我希望能得到AI分析的问题类型和可能原因，以便减少人工排查时间

**验收标准**：
- [ ] 输出问题类型（如连接超时、DNS解析失败）
- [ ] 列出Top 3可能原因
- [ ] 评估风险等级（P0/P1/P2）
- [ ] AI分析结果以JSON格式输出

**测试用例**：
1. 连接超时日志 → 识别为连接超时
2. DNS失败日志 → 识别为DNS解析失败
3. 正常日志 → 识别为无问题
4. API调用超时 → AI调用失败时有明确提示

---

### US-004: 生成诊断报告

**用户故事**：
> 作为运维工程师，我希望能生成诊断报告，以便分享给团队成员

**验收标准**：
- [ ] 生成Markdown格式报告
- [ ] 包含文件信息（文件名、大小、格式）
- [ ] 包含日志概览（总行数、错误行数、错误率）
- [ ] 包含AI分析结果（问题类型、可能原因、风险等级）
- [ ] 报告自动保存到reports/目录
- [ ] 报告文件名包含时间戳

**测试用例**：
1. 分析单个日志 → 生成单个报告
2. 分析多个日志 → 生成汇总报告
3. reports/目录不存在 → 自动创建

---

### US-005: 获取排查命令

**用户故事**：
> 作为运维工程师，我希望能获得可执行的排查命令，以便快速验证诊断结果

**验收标准**：
- [ ] 至少提供3类命令（网络连通性、端口检查、DNS解析）
- [ ] 命令格式包含说明和示例
- [ ] 不包含危险命令（如rm、重启服务）
- [ ] 命令适用于Linux系统

**测试用例**：
1. 连接超时问题 → 提供ping、traceroute等命令
2. DNS失败问题 → 提供dig、nslookup等命令
3. 端口不可达问题 → 提供ss、netstat等命令

---

### US-006: 简单易用的CLI

**用户故事**：
> 作为开发工程师，我希望命令行界面简单易用，以便快速上手

**验收标准**：
- [ ] 核心命令清晰（analyze、--log、--format、--model）
- [ ] 参数验证完善（缺失必需参数时提示）
- [ ] 错误提示友好（说明原因和解决方法）
- [ ] 使用彩色输出（Rich库）

**测试用例**：
1. 执行`netdiagnoser analyze --log test.log` → 成功分析
2. 执行`netdiagnoser analyze` → 提示缺少--log参数
3. 执行`netdiagnoser --help` → 显示帮助信息
4. 执行`netdiagnoser --version` → 显示版本号

---

### US-007: 详细的帮助文档

**用户故事**：
> 作为开发工程师，我希望有详细的帮助文档，以便了解如何使用工具

**验收标准**：
- [ ] 每个命令有help说明
- [ ] 提供使用示例
- [ ] 说明支持的日志格式（nginx、haproxy、syslog）
- [ ] README.md包含快速开始指南

**测试用例**：
1. 查看主命令帮助 → 显示所有子命令
2. 查看analyze命令帮助 → 显示所有参数
3. 查看README.md → 包含安装和使用说明

---

### US-008: 批量分析日志

**用户故事**：
> 作为开发工程师，我希望支持批量分析多个日志，以便提高效率

**验收标准**：
- [ ] 支持--log参数多次
- [ ] 生成汇总报告
- [ ] 批量进度显示

**测试用例**：
1. 分析2个日志文件 → 生成汇总报告
2. 分析5个日志文件 → 显示批量进度
3. 其中一个文件不存在 → 跳过并提示，继续处理其他文件

---

### US-009: 扩展日志格式

**用户故事**：
> 作为架构师，我希望支持扩展更多日志格式，以便适应不同场景

**验收标准**：
- [ ] 日志解析模块可扩展（插件式设计）
- [ ] 新增格式不修改核心逻辑
- [ ] 文档说明如何扩展

**测试用例**：
1. 新增Apache日志解析器 → 不影响现有解析器
2. 测试新解析器 → 能正确解析Apache日志

---

### US-010: 切换AI模型

**用户故事**：
> 作为架构师，我希望支持切换不同的AI模型，以便对比效果

**验收标准**：
- [ ] 支持--model参数
- [ ] 支持glm-4.7和glm-5.0
- [ ] 配置文件可设置默认模型
- [ ] 不支持的模型提示错误

**测试用例**：
1. 使用--model glm-4.7 → 调用GLM-4.7
2. 使用--model glm-5.0 → 调用GLM-5.0
3. 使用--model unknown → 提示不支持的模型
