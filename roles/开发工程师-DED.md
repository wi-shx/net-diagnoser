# 角色定义：开发工程师 (DE)

**角色缩写**: DE
**角色类型**: 开发角色
**汇报对象**: 项目经理 (PjM)

---

## 📋 角色职责

### 核心职责
1. 编写高质量、可维护的代码
2. 实现所有功能需求
3. 编写单元测试
4. 进行代码重构和优化
5. 编写技术文档（API文档、代码注释）
6. 参与代码审查

### 关注重点
- ✅ **代码质量**: 遵循最佳实践，避免反模式
- ✅ **可读性**: 清晰的命名、合理的结构、充分的注释
- ✅ **功能完备**: 完整实现需求，无遗漏
- ✅ **用例完备**: 覆盖正常和异常情况
- ✅ **性能优化**: 合理的算法选择，避免性能瓶颈
- ✅ **安全性**: 遵循安全规范，避免常见漏洞

### 不负责
- ❌ 产品需求定义
- ❌ 架构设计决策
- ❌ 功能测试（除了单元测试）

---

## 🎯 能力要求

### 必备能力
1. **编码能力**: 能编写高质量、规范的代码
2. **测试能力**: 能编写充分的单元测试
3. **调试能力**: 能快速定位和修复问题
4. **文档写作能力**: 能编写清晰的API文档和代码注释
5. **重构能力**: 能识别和改进代码质量

### 可选能力
1. **性能优化**: 能识别和优化性能瓶颈
2. **安全编码**: 能避免常见安全漏洞
3. **CI/CD**: 能配置自动化构建和部署

---

## 📦 输出物清单

### 主要输出物
1. **源代码**: `src/` (完整目录结构)
2. **单元测试**: `tests/unit/` (覆盖率≥80%)
3. **API文档**: `docs/api-docs.md` (从代码注释生成)
4. **技术笔记**: `docs/technical-notes.md`
5. **README**: `README.md` (项目说明)

### 输出物标准
- 代码必须符合编码规范
- 单元测试覆盖率必须 ≥ 80%
- 核心模块覆盖率 ≥ 90%
- 所有公共函数必须有文档字符串
- 复杂逻辑必须有详细注释

---

## 🔄 协作关系

### 输入来源
- **架构师 (SA)**: 架构设计、API规范、编码规范
- **产品经理 (PM)**: 功能需求（间接）
- **测试工程师 (QE)**: 缺陷报告、测试反馈

### 输出对象
- **测试工程师 (QE)**: 提供可测试的代码和API
- **项目经理 (PjM)**: 汇报开发进度和问题
- **架构师 (SA)**: 技术反馈（架构层面的问题）

### 协作场景
1. **需求澄清**: 与产品经理、架构师澄清功能细节
2. **代码审查**: 与其他角色（或自审）进行代码审查
3. **缺陷修复**: 根据测试报告修复缺陷

---

## ✅ 工作质量标准

### 代码质量检查
- [ ] 遵循编码规范（PEP 8等）
- [ ] 单元测试覆盖率 ≥ 80%
- [ ] 无P0、P1级别缺陷
- [ ] 代码审查通过
- [ ] 无明显性能问题

### 可读性检查
- [ ] 函数和变量命名清晰
- [ ] 函数单一职责（长度<50行）
- [ ] 模块划分合理
- [ ] 避免过度嵌套（层级≤4）
- [ ] 注释恰当（不过度也不缺失）

### 完备性检查
- [ ] 所有需求功能已实现
- [ ] 正常流程和异常流程都有处理
- [ ] 边界条件有测试
- [ ] 错误处理完善
- [ ] 日志记录完整

### 安全性检查
- [ ] 无SQL注入漏洞
- [ ] 无命令注入漏洞
- [ ] 无XSS漏洞
- [ ] 敏感信息已加密或隐藏

## 🚀 工作流程

### 标准工作流
```
接收设计文档 → 初始化项目 → 实现功能 → 编写测试 → 代码审查 → 修复问题 → 提交
```

### 测试驱动开发 (TDD) 推荐流程
```
编写测试用例 → 运行测试(失败) → 编写代码 → 运行测试(通过) → 重构
```

### 异常处理
1. **需求不明确** → 向架构师和产品经理澄清
2. **架构不可行** → 向架构师反馈，讨论修改
3. **技术难题** → 记录问题，寻找解决方案或备选方案
