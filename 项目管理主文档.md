## 一、项目概述

### 1.1 项目目标

构建一个AI驱动的网络故障诊断工具，支持日志上传、智能分析和诊断报告生成。

### 1.2 核心功能（MVP）

1. 支持多种日志格式解析（nginx、haproxy、syslog等）
2. AI分析错误模式并给出诊断建议
3. 生成结构化的诊断报告
4. 提供可执行的排查命令建议

### 1.3 SMART目标

- **S (Specific)**: 明确的功能范围和技术指标
- **M (Measurable)**: 测试覆盖率≥80%，支持≥3种日志格式
- **A (Attainable)**: 技术已验证，成本可控
- **R (Relevant)**: 提升运维效率，降低人工成本
- **T (Time-bound)**: MVP 1周完成

## 四、协作机制

### 4.1 角色切换规则

#### 切换时机
1. 当前角色所有任务完成
2. 所有验收标准通过
3. 下一角色有足够输入

#### 切换格式
```markdown
---
**角色切换**: [旧角色] → [新角色]
**触发原因**: [原因]
**时间**: [时间戳]
**上一角色交付**: [交付物清单]
**质量检查**: [✅通过 / ❌问题]
---

[进入新角色]
```

#### 切换检查清单
- [ ] 上一角色所有任务完成
- [ ] 交付物符合质量标准
- [ ] 下一角色输入完整
- [ ] 切换已记录

### 4.2 角色间协作关系图

```
用户
 ↓ ↓ ↓ ↓ ↓
 ┌─┬─┬─┬─┬─┐
PM SA DE QE PjM
 │ │ │ │ │
 └─PjM─────────┘
    ↑
  项目协调
```

### 4.3 典型协作场景

#### 场景1: 需求评审
```
PM → PjM: 组织评审会议
PM → SA: 技术可行性讨论
SA → PM: 技术建议反馈
```

#### 场景2: 开发中的问题
```
DE → SA: 架构问题反馈
SA → DE: 架构调整建议
DE → QE: 测试进度同步
```

#### 场景3: 缺陷修复
```
QE → DE: 提交缺陷报告
DE → QE: 修复进度反馈
QE → DE: 回归测试验证
```
